## TL;DR

This release brings `openclaw-xiaozhi-plugin` to feature parity with the
upstream [`xiaozhi-esp32-server`](https://github.com/78/xiaozhi-esp32-server)
across all 12 identified gaps in **VAD, ASR, audio accumulation, TTS queueing,
text cleaning, state machine, OAuth, error handling, and observability**.

All 12 new features are **opt-in via 8 grayscale flags** (default: all off),
preserving 100% V2 #5 (auth-disabled) and V2 #6.1 (Bearer token) compatibility.
Production users can incrementally enable each flag and roll back at any time.

## 🔒 Compatibility

**100% backward compatible.** The existing 184-test baseline continues to pass
unchanged. Every new feature is gated by a grayscale flag that **defaults to
`false` / `undefined`**.

- ✅ V2 #5 (auth disabled) hot path — **0 lines changed**
- ✅ V2 #6.1 (Bearer token) hot path — **0 lines changed**
- ✅ Existing `package.json` consumers — `onnxruntime-node` is **dynamic-imported**, not bundled
- ✅ Roll-back path — disable flags + restart, no schema migration needed

### 8 Grayscale Flags

All flags live under `channels.xiaozhi.*` in your gateway config. All default
to `false`.

| Flag | Default | Enable when… |
|------|:-------:|--------------|
| `useTtsPipeline`     | `false` | You want sentence-level streaming TTS with bounded memory |
| `useTextCleaner`     | `false` | Your LLM emits Markdown / `replace_words` artifacts mid-stream |
| `useMetrics`         | `false` | You want `/api/xiaozhi/metrics` for Prometheus / debugging |
| `useSileroVad`       | `false` | You need server-side VAD and have the Silero ONNX model installed |
| `useStreamingAsr`    | `false` | You want partial ASR results via `OnlineRecognizer` |
| `useAccumulatePcm`   | `false` | You want opus-decoded int16 PCM stored on the session |
| `useMultiFlagState`  | `false` | You want the 4 new `clientHaveVoice` / etc. session flags |
| `useOAuth`           | `false` | You have an external Authorization Server with `client_credentials` |
| `useRetry`           | `false` | You want exponential backoff on TTS / ASR / model-load failures |

> Note: `useTtsPipeline` / `useTextCleaner` / `useMetrics` are read at runtime
> via the corresponding `getUse*` accessor in
> [`src/api.ts`](https://github.com/hqgaofeng/openclaw-xiaozhi-plugin/blob/v0.4.0/src/api.ts#L83-L141).

## 🚀 Installation / Upgrade

For existing users on a deployed gateway:

```bash
cd /root/projects/openclaw-xiaozhi-plugin
git pull
git checkout v0.4.0
npm install   # pulls onnxruntime-node ^1.20.0 (dynamic import, not bundled)
sudo systemctl --user restart openclaw-gateway

# Sanity check: with all flags off, metrics endpoint should 404
curl -i http://localhost:18790/api/xiaozhi/metrics
# expected: HTTP/1.1 404 Not Found  (useMetrics is false)
```

To opt into a feature, add the relevant flag to your gateway config and
restart — no other code change required.

For a fresh install, follow the standard
[`README.md`](https://github.com/hqgaofeng/openclaw-xiaozhi-plugin/blob/v0.4.0/README.md)
instructions; v0.4.0 is a drop-in replacement for v0.3.x.

## ⚠️ Known Issues / Caveats

- **Silero VAD model is not bundled.** You must manually download
  `silero_vad.onnx` (v4) to `/opt/xiaozhi-plugin/models/silero/` on the VPS.
  See [`docs/architecture.md`](https://github.com/hqgaofeng/openclaw-xiaozhi-plugin/blob/v0.4.0/docs/architecture.md)
  for the exact path and wget one-liner. CI tests use a mocked
  `onnxruntime-node`, so they pass without the model.
- **`onnxruntime-node` is OS/arch specific.** `npm install` will compile
  / fetch a native binary for your platform. Make sure your deployment
  target matches your dev target (or use `prebuild-install`).
- **OAuth depends on an external Authorization Server.** With `useOAuth: false`
  (the default), the gateway falls through to V2 #6.1 Bearer auth — the
  behavior your existing V2 deployment already uses. There is no built-in
  IdP in this release.
- **All flags default `false`.** Nothing in v0.4.0 turns itself on. You must
  explicitly enable each flag you want.
- **5 commits, not squashed.** This was a release-engineering choice — keeping
  each batch as its own commit makes it trivial to `git revert` a single
  batch (e.g. `git revert e10f978`) without dragging the others along.
  See [`CHANGELOG.md`](https://github.com/hqgaofeng/openclaw-xiaozhi-plugin/blob/v0.4.0/CHANGELOG.md)
  for the squashed narrative.
- **`vitest` timeout raised to 30s** in
  [`vitest.config.ts`](https://github.com/hqgaofeng/openclaw-xiaozhi-plugin/blob/v0.4.0/vitest.config.ts)
  to accommodate the OAuth + retry test suites. The default 5s was too tight
  for the backoff-math tests.

## 🛣️ Roadmap (v0.5.0 and beyond)

Allen (项目负责人) will re-prioritize after the v0.4.0 rollout stabilizes.
Tentative candidates, in no particular order:

- **Multi-device P2P communication** — direct ESP32 ↔ ESP32 audio routing for
  mesh deployments
- **MCP tool marketplace** — discoverable, versioned tool registry
  (replaces the current static `manifest.json` tool list)
- **LLM streaming — multi-provider** — currently MiniMax-only; add OpenAI /
  Anthropic / local Ollama providers with a shared streaming interface
- **OTA incremental updates** — diff-based firmware delivery to ESP32 clients
  via the existing WebSocket channel
- **Native wake-word model** — replace the current RMS-gated path with a
  on-device keyword spotter (openWakeWord or Porcupine)
- **Prometheus exporter** — promote `/api/xiaozhi/metrics` to a proper
  `/metrics` text-format endpoint

If any of these are blockers for you, please open an issue or ping Allen
directly — the v0.5.0 scope is still open.

## 2. 部署步骤

> **工作目录**: 以下所有命令均在 `/root/projects/openclaw-xiaozhi-plugin` 下执行。
> 推荐先开一个 tmux 会话, 防止 SSH 断线杀掉 `npm install`。
> 预计耗时: 网络正常时 5-8 分钟, Silero 模型下载另算 1-2 分钟。

### 2.1 拉最新代码 + 装新依赖

```bash
cd /root/projects/openclaw-xiaozhi-plugin

# 1) 拉取 v0.4.0 收口 commit (已在 origin, 此处保险)
git fetch --tags origin
git status
# 期望: "Your branch is up to date with 'origin/main'."
# 期望 HEAD: d665bd2 feat(plug): v0.4.0 — 12 项官方对齐重构收口 (batch 5)

# 2) 装新依赖 (onnxruntime-node ^1.20.0 是 v0.4.0 新增)
npm install
# 期望: added 1 package, 0 vulnerabilities (onnxruntime-node 走 prebuilt binary)

# 3) 自检: 编译干净 + 330 tests 全绿
npx tsc --noEmit
npm test
# 期望: Test Files  25 passed (25)
#       Tests       330 passed (330)
```

> **不通过怎么办**: 报错 `EACCES` / `ENOSPC` → 看 §5.3; 测试红 → 看 §6。

### 2.2 下载 Silero VAD 模型 (可选, 建议下)

```bash
# 1) 建模型目录 (vad-silero.ts 默认路径)
sudo mkdir -p /opt/xiaozhi-plugin/models/silero/
sudo chown -R root:root /opt/xiaozhi-plugin

# 2) 下载 silero_vad v4 ONNX (100 MiB)
#    官方源: https://github.com/snakers4/silero-vad/raw/master/src/silero_vad/data/silero_vad.jit
#    ⚠️ 注意是 .jit (TorchScript), ONNX 在 v4 release artifacts
#    推荐 ONNX 源 (Release v4.0): 
curl -L -o /opt/xiaozhi-plugin/models/silero/silero_v4.onnx \
  "https://github.com/snakers4/silero-vad/releases/download/v4.0/silero_v4.onnx"

# 3) 校验
ls -lh /opt/xiaozhi-plugin/models/silero/silero_v4.onnx
# 期望: ~100M, date 今天的

# 4) (可选) sha256 校验
sha256sum /opt/xiaozhi-plugin/models/silero/silero_v4.onnx
```

> ⚠️ **不要** `npm install silero-vad-model` 这种包, 不存在, 必走 curl/GitHub Release。
> ⚠️ **不进** `node_modules/` — 这是 v0.4.0 设计上把模型拆出 npm 的原因, 镜像大小可控。

### 2.3 重启 gateway

```bash
# 1) 优雅停
openclaw gateway stop
# 或: kill -TERM 1765769 (查 ps aux)

# 2) 确认端口 18789 已释放
sleep 2
ss -tlnp | grep ':18789' || echo "port 18789 free ✓"

# 3) 拉起
openclaw gateway start

# 4) 查 PID
sleep 2
ps aux | grep "openclaw.*gateway" | grep -v grep
# 期望: 1 行, 新 PID

# 5) tail 日志, 看 plugin 是否加载
journalctl -u openclaw-gateway -n 50 --no-pager 2>/dev/null \
  || tail -50 ~/.openclaw/logs/gateway.log
# 期望: 含 "loaded plugin: xiaozhi (v0.4.0)" 或类似字样
```

> ⚠️ **不要** 用 `kill -9` — gateway 会有未 flush 的 session state, 可能卡 OTA 下次启动。
> ⚠️ **停服窗口**: 30-60 秒, esp32 端会断 WS 重连, 不影响硬件。

## 3. 灰度开关开启顺序

> **核心原则**: 一项开一项验一项, 一次只开 1 个 flag, 中间留 5-10 分钟观察。
> 所有 flag 默认 `false` / `undefined`, v0.4.0 与 v0.3.0 行为完全一致。
> 配置位置: `~/.openclaw/openclaw.json` → `channels.xiaozhi.<flag>`

### 3.1 推荐开启顺序总表

| 序 | flag | 风险 | 内存影响 | 关键依赖 | 推荐前置观察 | 回滚 (单 flag) | 验证命令 |
|---|---|---|---|---|---|---|---|
| 1 | `metricsEnabled` | 🟢 极低 (纯观测) | +5 MiB | 无 | 5 分钟 | 改回 `false` 重启 | `curl /api/xiaozhi/metrics` |
| 2 | `useRetry` | 🟢 低 (retry 包了才生效) | +1 MiB | 无 | 5 分钟 | 改回 `false` 重启 | `journalctl` 看 retry 计数 |
| 3a | `useStreamingTts` | 🟡 中 (TTS 行为改变) | +10 MiB | `useTextCleaner` | 30 分钟 | 改回 `false` 重启 | `journalctl` 看 pipeline 队列 |
| 3b | `useTextCleaner` | 🟡 中 (与 TTS 捆绑) | +2 MiB | 推荐同 #3a | (同 #3a) | 改回 `false` 重启 | `journalctl` 看 cleaner hit |
| 4 | `useMultiFlagState` | 🟡 中 (session flag 改变) | +1 MiB | 无 | 15 分钟 | 改回 `false` 重启 | 单元测试覆盖 |
| 5 | `useAccumulatePcm` | 🟠 中高 (opus decode 路径改变) | **-30 MiB** ⚡ | 无 | 30 分钟 | 改回 `false` 重启 | `journalctl` 看 first-partial 延迟 |
| 6 | `useStreamingAsr` | 🔴 高 (ASR 引擎改变) | +200 MiB | 模型已在 `/opt/...` | 1 小时 | 改回 `false` 重启 | esp32 端到端 ASR 准确率 |
| 7 | `useSileroVad` | 🔴 高 (VAD 模型 + 内存) | **+150 MiB** | `silero_v4.onnx` 已下 | 1 小时 | 改回 `false` 重启 | `journalctl` 看 VAD 命中 |
| 8 | `useOAuth` | 🔴 高 (鉴权链路改变) | +5 MiB | **外部 Auth Server** | 2 小时 | 改回 `false` 重启 | esp32 端 OAuth 流程 |

> 🟢 = 纯观测 / 包装层, 不动业务路径
> 🟡 = 改业务路径但有单元测试覆盖
> 🟠 = 改核心数据流 (opus decode)
> 🔴 = 改核心算法 / 引入外部依赖 / 内存增量 > 100 MiB

### 3.2 详细开启步骤 (示例: flag #1 metricsEnabled)

```bash
# 1) 备份 openclaw.json
cp ~/.openclaw/openclaw.json ~/.openclaw/openclaw.json.bak.pre-$(date +%Y%m%d-%H%M%S)

# 2) 翻 flag
openclaw config set channels.xiaozhi.metricsEnabled true

# 3) 校验
openclaw config get channels.xiaozhi.metricsEnabled
# 期望: true

# 4) 重启 gateway
openclaw gateway restart

# 5) 验证 (见 §4.1)
curl -sS http://127.0.0.1:18790/api/xiaozhi/metrics | head -c 500
# 期望: JSON 200, 含 counter / histogram / gauge
```

### 3.3 开启节奏建议

| 时段 | 操作 | 备注 |
|---|---|---|
| **Day 1 (周一)** | 升级 v0.4.0 + 步骤 2.1-2.8 全跑 | 不开任何 flag, 验证 baseline |
| **Day 1 晚** | 翻 #1 `metricsEnabled` | 跑 §4.1, 看 prometheus JSON |
| **Day 2 早** | 翻 #2 `useRetry` | 跑 §4.2 |

| **Day 3 早** | 翻 #5 `useAccumulatePcm` | 跑 §4.5, esp32 实测首字延迟 |

## 4. 每步验证命令

> 每条命令执行前确认 gateway 已 restart (翻 flag 后必须)。

### 4.1 metricsEnabled=true

```bash
# 1) 路由现在返 200 JSON
curl -sS http://127.0.0.1:18790/api/xiaozhi/metrics | head -c 800
# 期望: JSON, 含 "counters" / "histograms" / "gauges" 字段

# 2) 触发一次 esp32 消息 (唤醒 or 按键), 再看计数

  | grep -oE '"xiaozhi_[a-z_]+_total":[0-9]+' | head -10
# 期望: 多个 counter 出现, 值 > 0

# 3) 看内存变化
ps -o rss= -p $(pgrep -f "openclaw.*gateway")
# 期望: 略增 (~5 MiB, vs baseline)
```

### 4.2 useRetry=true

```bash
# 1) 故意断网 30 秒 (sherpa-onnx 模型重试路径)
#    或: iptables -I OUTPUT -p tcp --dport 443 -j DROP
# 2) esp32 端触发一次 TTS 调用
# 3) 看日志里的 retry 字样
journalctl -u openclaw-gateway --since "1 minute ago" --no-pager \
  | grep -iE "retry|backoff" | tail -10
# 期望: "retry attempt 1/3" / "backoff 200ms" 等

# 4) 恢复网络
iptables -D OUTPUT -p tcp --dport 443 -j DROP
```

## 5. 回滚方案

### 5.1 单 flag 回滚 (最快, 30 秒)

```bash
# 1) 翻回 false
openclaw config set channels.xiaozhi.<flag> false

# 2) 重启
openclaw gateway restart

# 3) 验证
journalctl -u openclaw-gateway -n 30 --no-pager \
  | grep -E "<flag>|xiaozhi" | tail -10
# 期望: flag 不再出现在日志
```

### 5.2 整个 v0.4.0 回滚 (回到 v0.3.7, 2 分钟)

```bash
# 1) 停 gateway
openclaw gateway stop

# 2) 回代码到上一个稳定 tag
cd /root/projects/openclaw-xiaozhi-plugin
git checkout v0.3.7
# 注: v0.3.7 是占位 tag, 实际可能叫 v0.3.6 / v0.3.0; 取最后一个稳定 tag
git tag | tail -5
# 改用:
git checkout v0.3.0   # v0.3.0 已验证 live

# 3) 清 v0.4.0 装的 native binding (避免 ABI 不兼容)
rm -rf node_modules/onnxruntime-node
npm ci

# 4) (可选) 删 Silero 模型 (省 100 MiB, 之后可重下)
# 保留也行, v0.3.0 不会读它

# 5) 拉起
openclaw gateway start

# 6) 验证 OTA 仍 200
curl -sS https://jarvis.beallen.top/api/xiaozhi/ota/ \
  -X POST -H "Content-Type: application/json" -d '{}' | head -c 200
# 期望: 200, JSON
```

> ⚠️ **回滚后**: openclaw.json 里 v0.4.0 灰度 flag 还在, 但 plugin 不认, 启动会被 strip 掉 (M3.4g 修复), 无副作用。

### 5.3 内存不够 / OOM 怎么办

| 现象 | 原因 | 处置 |
|---|---|---|
| gateway 启动后 5 分钟内被 OOM killer | `onnxruntime-node` 加载 + Silero 模型 | §5.3.1 |
| `useStreamingAsr` 翻 true 后 OOM | sherpa-onnx streaming 模型 200+ MiB | §5.3.2 |
| esp32 大并发 (5+ 设备) 顶到 swap | 每个 session 各持 ASR/TTS state | §5.3.3 |

#### 5.3.1 native binding + Silero OOM

```bash
# 1) 立即停
openclaw gateway stop

# 2) 看 dmesg
dmesg | grep -i "killed process" | tail -5
# 期望: "Killed process XXXX (node) total-vm:..."

# 3) 选项 A: 翻回 false
openclaw config set channels.xiaozhi.useSileroVad false
# 选项 B: 不下 Silero 模型 (删文件)
rm /opt/xiaozhi-plugin/models/silero/silero_v4.onnx

# 4) 拉起

# 5) 监控 5 分钟
watch -n 5 'free -h && ps -o rss= -p $(pgrep -f "openclaw.*gateway")'
```

## 6. 已知问题 / 限制

### 6.1 模型 / 依赖不在 npm 包内

| 项 | 原因 | 处置 |
|---|---|---|
| Silero VAD 模型 (`silero_v4.onnx`) | 100 MiB, 不进 npm | §2.2 手动下到 `/opt/xiaozhi-plugin/models/silero/` |
| onnxruntime-node native binding | 50 MiB 平台相关 | `npm install` 时自动拉 prebuilt; ARM 平台需编译 |
| sherpa-onnx ASR 模型 | v0.3.0 就有, 不在 v0.4.0 改动 | 路径见 `channels.xiaozhi.asr.options.model` |

### 6.2 onnxruntime-node 加进 dependencies

- 升级 v0.4.0 后 `npm install` 会拉 prebuilt binary
- 阿里云镜像如果没 onnxruntime-node, 走官方 npmjs.org:
  ```bash
  npm config get registry
  # 期望: https://registry.npmjs.org/
  # 切换: npm config set registry https://registry.npmjs.org/
  ```
- 如果 VPS 装的是 musl libc (Alpine), prebuilt 不通用, 需装 `python3 make g++` 后从源码编译

### 6.3 OAuth 部署依赖外部 Authorization Server

- v0.4.0 翻 `useOAuth=true` **必须**有可用的 Auth Server
- 没配就翻 `true` → esp32 全连不上, 表现同 401 Unauthorized
- 临时方案: 保持 `useOAuth=false`, 用 V2 #6.1 的 `XIAOZHI_AUTH_TOKEN` 单 token (M3.4g 路径)
- 长期方案: 搭本仓的 xiaozhi-oauth-server (待立项) 或用 Auth0/Keycloak

### 6.4 灰度 flag 持久化问题

- `openclaw config set` 改的是 `~/.openclaw/openclaw.json`
- 备份文件 `*.bak.*` 不要手删, 至少留最近 3 份
- 如果 openclaw 自动 reload 配置 (需查 `openclaw config validate`), 不用重启, 否则 §2.3 步骤必须

### 6.5 单元测试必须先过

- §2.1 的 `npm test` 是 **唯一合法交付门** (Allen 红线 #1)
- 如果测试红:
  - 不要动 `src/` (本手册不涉及)
  - 回滚 commit: `git revert HEAD` 或 `git reset --hard v0.3.0`
  - 报告 red case 给上游

### 6.6 nginx 反代 18790 端口

- 公网路径固定: `https://jarvis.beallen.top/api/xiaozhi/{ota,metrics}/`
- WebSocket 路径: `wss://jarvis.beallen.top/xiaozhi/v1/`
- ⚠️ **不要改 nginx 配置** — v0.4.0 没改 path, esp32 固件无需重刷

### 6.7 已知未修小坑

| 坑 | 影响 | 工作区 |
|---|---|---|
| onnxruntime-node 在 musl libc (Alpine) 需从源码编译 | 5+ 分钟 | `npm install` 时关注 |
| Metrics JSON 无 Prometheus 文本格式导出 | 不能直接接 Grafana | V0.4.1 |
| OAuth token store 内存态, 重启丢 | 设备需重新授权一次 | V0.4.1 改 sqlite |
| `useStreamingAsr` 中文 80%+, 英文 bpe fallback 弱 | 同 V2 #1 sherpa-onnx 局限 | 不可修, 模型限制 |
